# Install the CLI

The CLI is the canonical Diffmint surface. It is where reviews, explain flows, test generation,
history checks, and diagnostics are expected to start.

## What this setup gives you

- Review commands that stay close to your local git workflow
- Browser-based device auth tied to the correct workspace
- Diagnostics that verify runtime, auth, and provider readiness
- A stable foundation for both terminal and VS Code usage

## 1. Install dependencies

```bash
pnpm install
```

## 2. Sign in and approve the machine

```bash
dm auth login
```

The browser flow lands on a dedicated device approval page. Sign in, switch to the correct
workspace if needed, approve the device session, then return to the terminal. Approval is required
before the CLI can fetch workspace bootstrap or sync review history.

## 3. Verify provider and machine health

```bash
dm config set-provider qwen
dm doctor
```

If `qwen` is available on your machine and your provider credentials are configured, review runs
will use Qwen headless mode automatically. Otherwise Diffmint stays in scaffold mode so onboarding
and local testing still work.

## 4. Run the commands users depend on most

```bash
dm review --staged
dm review --base origin/main
dm review --files src/app/page.tsx src/lib/review.ts
dm explain src/app/page.tsx
dm tests src/lib/review.ts
```

These commands cover the day-to-day workflow:

- `review` for code review from the terminal
- `explain` for targeted explanation on a file or surface
- `tests` for test scaffolding and coverage help

## What good installation looks like

You are done when:

- `dm auth login` completes without a pending approval step
- `dm doctor` reports a healthy local environment
- `dm review` can run without reconfiguring the machine

See the full [CLI reference](/docs/cli/reference) for flags, output modes, and scripting support.

For containerized local development with hot reload, see [Docker development](/docs/getting-started/docker-development).
