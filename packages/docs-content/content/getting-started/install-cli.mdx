# Install the CLI

The CLI is the canonical Diffmint surface. It is where reviews, explain flows, test generation,
history checks, and diagnostics are expected to start.

## What this setup gives you

- Review commands that stay close to your local git workflow
- Browser-based device auth tied to the correct workspace when you need sync
- Local Codex, Antigravity, or API-key auth when you want BYOK and zero hosted provider keys
- Diagnostics that verify runtime, auth, and provider readiness
- A stable foundation for both terminal and VS Code usage

## 1. Install the CLI

```bash
npm install -g @unpijs/dm
```

If you are working on the Diffmint monorepo itself, use `pnpm install` in the repo root instead.

## 2. Choose how this machine authenticates

Use the flow that matches your rollout stage:

```bash
dm auth login
dm auth login codex
dm auth login antigravity
dm auth login api
```

- `dm auth login` keeps the control-plane device flow for workspace bootstrap, policy sync, and history upload.
- `dm auth login codex` uses local Codex auth from `codex login`.
- `dm auth login antigravity` keeps Antigravity auth on the user machine.
- `dm auth login api` reads a local provider key from `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `QWEN_API_KEY`, or `DASHSCOPE_API_KEY`.

If you need the remote workspace flow, approve the device in your browser:

```bash
dm auth login
```

The browser flow lands on a dedicated device approval page. Sign in, switch to the correct
workspace if needed, approve the device session, then return to the terminal. Approval is required
before the CLI can fetch workspace bootstrap or sync review history. The BYOK flows above do not
upload provider keys to Diffmint.

## 3. Verify provider and machine health

```bash
dm doctor
```

`dm doctor` reports:

- the selected provider, auth mode, and model
- whether Codex, Antigravity, Qwen, or a local API key are available
- whether the machine is connected to a control plane or running local-only

You can override the defaults later with:

```bash
dm config set-provider codex
dm config set-model gpt-5-codex
```

## 4. Run the commands users depend on most

```bash
dm review --staged
dm review --base origin/main
dm review --files src/app/page.tsx src/lib/review.ts
dm explain src/app/page.tsx
dm tests src/lib/review.ts
```

These commands cover the day-to-day workflow:

- `review` for code review from the terminal
- `explain` for targeted explanation on a file or surface
- `tests` for test scaffolding and coverage help

## What good installation looks like

You are done when:

- `dm auth login` completes without a pending approval step
- `dm doctor` reports a healthy local environment
- `dm review` can run without reconfiguring the machine

See the full [CLI reference](/docs/cli/reference) for flags, output modes, and scripting support.

For containerized local development with hot reload, see [Docker development](/docs/getting-started/docker-development).
