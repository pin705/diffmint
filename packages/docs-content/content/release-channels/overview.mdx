# Release Channels

Release channels let teams move quickly without making every user absorb the same level of change
at the same time.

## Channel model

| Channel   | Intended audience | Recommended use                                                |
| --------- | ----------------- | -------------------------------------------------------------- |
| `stable`  | Most users        | Default production channel with the highest rollout confidence |
| `preview` | Early adopters    | Validate upcoming changes before they become the default       |
| `canary`  | Internal teams    | Fast feedback on packaging, install, and runtime changes       |

## Release manifest expectations

Each release manifest should include:

- version
- checksum
- download target
- release notes summary
- channel assignment

Users should not need to read raw internal changelog entries just to understand whether an update
is safe for them.

For production rollouts, Diffmint can also sign release manifests before returning them from the
client API. When `DIFFMINT_RELEASE_SIGNING_PRIVATE_KEY` is configured, manifests include a stable
`signature` envelope with:

- `algorithm` set to `ed25519`
- `keyId` for key rotation and rollout tracking
- `signedAt` anchored to the manifest release timestamp
- `value` as the base64 signature over the canonical manifest payload

## Recommended rollout policy

1. Ship to `canary` first for packaging and install validation.
2. Promote to `preview` when runtime and upgrade behavior look healthy.
3. Move to `stable` only after support, rollback, and release-note quality are all ready.
